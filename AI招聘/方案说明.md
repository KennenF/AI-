# RecruitFlow AI：招聘数据自动化记录与同步方案

## 1. 场景痛点

现有招聘流程中，HR 主要通过企业微信、企业微信群和腾讯在线文档协作。候选人信息、面试进度、跟进时间、面试反馈等数据需要人工记录和更新，容易出现：

- 群聊信息分散，候选人状态难追踪。
- 手工录入耗时，重复劳动多。
- 腾讯文档更新滞后，数据口径不统一。
- 跟进提醒依赖人工记忆，候选人容易遗漏。
- 招聘过程缺少实时看板，管理者难以及时判断招聘进度。

## 2. 方案目标

在最小成本预算下，构建一个轻量 AI 招聘数据助理，实现：

- 自动读取或接收企业微信群招聘消息。
- 将自然语言消息结构化为候选人数据。
- 自动新增或更新候选人招聘阶段。
- 使用普通 JSON/CSV 文件完成 Demo 存储。
- 可视化展示招聘漏斗、渠道来源、待跟进、逾期提醒。
- 预留腾讯文档同步接口。
- 预留企业微信机器人推送接口。
- 预留大模型 API 抽取接口，后续可替换规则抽取器。
- 根据面试进展自动生成企业微信工作群提醒。

## 3. Demo 定位

当前 Demo 不接入大模型 API，使用规则抽取器作为 AI 小替代，保证没有密钥也能直接运行。

核心链路：

```txt
企业微信群消息 / HR 粘贴
        ↓
POST /api/wecom/inbound
        ↓
AIExtractor 统一抽取入口
        ↓
规则抽取器 / 后续大模型 API
        ↓
候选人去重与状态更新
        ↓
data/candidates.json
        ↓
招聘看板实时展示
        ↓
腾讯文档 Mock 同步 / 企微机器人面试进展提醒
```

## 4. 功能模块

### 4.1 消息采集

Demo 页面提供“消息采集”入口，HR 可粘贴企业微信群消息。

后端接口：

```txt
POST /api/messages/ingest
POST /api/wecom/inbound
```

其中 `/api/wecom/inbound` 是企业微信群实时消息入口，Demo 页面会展示回调地址和 Token，并提供“模拟企微群消息推送”按钮。点击后会模拟企业微信把群聊内容推送到系统，系统立即完成 AI 抽取、候选人入库和看板更新。

正式落地时，可由企业微信应用回调、会话存档、表单或内部转发入口将消息提交到该接口。普通企业微信群机器人 Webhook 更适合向群里发提醒，不适合作为读取群消息的唯一入口。

### 4.2 AI 结构化抽取

当前使用 `src/ruleExtractor.js` 实现规则抽取，识别字段包括：

- 姓名
- 手机号
- 岗位
- 来源渠道
- 招聘阶段
- 当前状态
- 面试官
- 下次跟进时间
- 备注
- 置信度
- 缺失字段

当前 Demo 的统一入口是 `src/aiExtractor.js`。它会读取配置决定走规则抽取还是大模型抽取：

```txt
mode=rule  -> src/ruleExtractor.js
mode=llm   -> src/llmExtractor.js
```

`src/llmExtractor.js` 已经预留大模型 API 请求结构、Prompt 和 JSON 输出约束。后续只需在配置中填写 endpoint、model 和 apiKey，即可将自然语言抽取能力切换到大模型。

如果大模型 API 未配置或调用失败，系统会自动回退到规则抽取器，保证 Demo 不会因为没有密钥而运行失败。

### 4.3 候选人去重与更新

去重规则：

- 手机号一致：判定为同一候选人，更新状态。
- 手机号缺失但姓名 + 岗位一致：判定为疑似同一候选人。
- 新消息会追加到候选人的历史记录中。
- 状态更新后标记为 `pending`，等待同步腾讯文档。

### 4.4 文件化数据存储

当前 Demo 采用普通文件，降低部署成本：

```txt
data/candidates.json
data/messages.json
data/sync_logs.json
data/push_logs.json
data/config.json
exports/candidates.csv
```

生产环境可将文件存储替换为 MySQL、PostgreSQL、MongoDB 或企业 ATS 系统数据库。

### 4.5 招聘看板

看板自动聚合文件数据，展示：

- 候选人总数
- 今日新增
- 待跟进人数
- 逾期未跟进人数
- 招聘阶段漏斗
- 来源渠道分布
- 最近候选人列表

### 4.6 腾讯文档同步预留

Demo 中点击“同步腾讯文档”会生成：

```txt
exports/candidates.csv
```

并写入：

```txt
data/sync_logs.json
```

正式落地时，将 Mock 同步替换为腾讯文档开放平台 API 调用即可。

### 4.7 企业微信机器人预留

Demo 中点击“测试推送”或“推送面试进展提醒”会写入：

```txt
data/push_logs.json
```

面试进展提醒会根据候选人的阶段和下次跟进时间生成，例如：

```txt
RecruitFlow AI 面试进展提醒
1. 张三 / Java 后端实习生：初面，2026-07-22，推进二面安排，请按时推进。
2. 李娜 / 大模型算法实习生：简历筛选，2026-07-17，跟进当前招聘进展，请按时推进。
```

正式落地时，将 Mock 推送替换为企业微信机器人 Webhook 请求，用于推送：

- 今日待跟进提醒
- 二面安排提醒
- 面试反馈提醒
- 逾期候选人提醒
- 招聘日报
- 同步失败告警

## 5. 最小成本策略

- 不依赖数据库，使用普通文件完成 Demo。
- 不接入大模型 API，先用规则抽取器保证可运行。
- 预留 LLMExtractor，后续只换抽取实现，不影响业务流程。
- 无需安装前端框架，页面使用原生 HTML/CSS/JS。
- 后端仅使用 Node.js 内置模块，无 npm 依赖。
- 企业微信和腾讯文档先走 Mock，保留真实接口替换点。

## 6. 后续升级路线

第一阶段：当前 Demo

- 规则抽取
- 文件存储
- Mock 同步
- Mock 推送
- 招聘看板

第二阶段：真实业务试点

- 接入企业微信机器人或应用回调。
- 接入腾讯文档开放平台 API。
- 引入大模型 API 提升复杂消息识别能力。
- 自动向企业微信工作群推送二面、HR 面、Offer 跟进等进展提醒。
- 增加 HR 人工确认队列。

第三阶段：生产化

- 文件存储替换为数据库。
- 增加权限控制和操作审计。
- 增加候选人生命周期分析。
- 对接 ATS、面试日历、邮件或短信系统。

## 7. 预期效果

- 减少 HR 手工录入候选人数据的时间。
- 降低群聊信息遗漏概率。
- 提升招聘状态更新时效。
- 让管理者实时看到招聘漏斗。
- 通过机器人提醒减少跟进遗漏。
- 为后续招聘流程自动化和智能化打基础。
